# سما (SEMA) — سیستم مدیریت اداری سازمان

سیستم جامع مدیریت اداری با کنترل سطح دسترسی نقش‌محور (RBAC)، ورود دومرحله‌ای با کد یکتا + OTP پیامکی، و کارتابل‌های جداگانه برای مدیریت، مالی، منابع انسانی، مدیر میانی و کارمندان.

## تکنولوژی‌ها

| بخش | تکنولوژی |
|---|---|
| API | Node.js 20+ / Bun — Express + Mongoose |
| فرانت‌اند | Next.js 16 (App Router) + Tailwind CSS 4 |
| دیتابیس | MongoDB |
| احراز هویت | OTP پیامکی + JWT در کوکی HttpOnly (access + refresh) |

## راه‌اندازی سریع

```bash
# ۱) دیتابیس (اگر MongoDB محلی ندارید)
docker compose up -d

# ۲) نصب وابستگی‌ها (از ریشه پروژه — workspaces)
npm install

# ۳) تنظیمات محیطی
cp apps/api/.env.example apps/api/.env
cp apps/web/.env.example apps/web/.env

# ۴) ساخت کاربر مدیر اولیه (کد یکتای ورود را چاپ می‌کند)
npm run seed

# ۵) اجرای API (پورت 4000)
npm run dev:api

# ۶) اجرای فرانت (پورت 3000) — در ترمینال دیگر
npm run dev:web
```

سپس به `http://localhost:3000` بروید، **کد یکتای** چاپ‌شده در مرحله seed را وارد کنید؛ کد OTP در حالت توسعه (`SMS_PROVIDER=console`) **در ترمینال API چاپ می‌شود**.

> اجرای API با Bun: `cd apps/api && bun src/index.js`

## گردش کار ورود

1. مدیر از «پنل مدیریت کاربران» کارمند را با شماره تلفن، سمت شغلی، نقش و سایر اطلاعات ثبت می‌کند — ثبت‌نام فقط از پنل مدیریت ممکن است.
2. سیستم یک **کد یکتا** (مثل `SEMA-AB2C-9XK4`) تولید می‌کند که مدیر به کارمند می‌دهد.
3. کارمند در صفحه ورود کد یکتا را وارد می‌کند ← کد **یک‌بارمصرف ۶ رقمی** به شماره او پیامک می‌شود (۵ دقیقه اعتبار، حداکثر ۵ تلاش).
4. با تایید OTP، توکن‌های JWT در کوکی HttpOnly ست می‌شوند.

## نقش‌ها و سطح دسترسی

| نقش | دسترسی‌ها |
|---|---|
| **ADMIN (مدیریت)** | مدیریت کاربران، مشاهده لاگ همه اتفاقات + دسترسی به همه بخش‌ها |
| **FINANCE (مالی)** | تعریف انواع درخواست مالی (وام، مساعده و...) + کارتابل بررسی درخواست‌ها |
| **HR (منابع انسانی)** | تعریف قابلیت‌ها (رزرو غذا، جلسه با مدیران و...) + کارتابل خدمات + تایید نهایی مرخصی |
| **MANAGER (مدیر میانی)** | دریافت گزارش‌کار تیم + تایید مرحله اول مرخصی تیم |
| **SITE_MANAGER (مدیر سایت)** | تایید مرخصی، وام و خرید کارمندانی که مدیر اصلی‌شان است |
| **GENERAL_MANAGER (مدیر کل)** | تایید مرخصی، وام و خرید — با دید روی همه درخواست‌های مرحله مدیر اصلی |
| **EMPLOYEE (کارمند)** | مشاهده قابلیت‌های فعال، ثبت درخواست مالی/خدمات، مرخصی (با امکان ویرایش/لغو)، ارسال گزارش‌کار |

نقش ADMIN به‌صورت خودکار به همه بخش‌ها دسترسی دارد. کنترل دسترسی در **سمت API** با middleware انجام می‌شود؛ منوی فرانت فقط جنبه نمایشی دارد.

## گردش کار مرخصی (چند تایید)

```
کارمند ثبت می‌کند ← مدیر میانی ← مدیر اصلی (سایت/کل) ← منابع انسانی (تایید نهایی)
```

- تایید مدیریتِ خود کارمند و منابع انسانی هر دو الزامی است.
- هر کارمند در تعریفش «مدیر میانی» و «مدیر اصلی» (مدیر سایت یا مدیر کل) دارد؛ مراحلی که تعریف نشده‌اند به‌صورت خودکار رد می‌شوند.
- کارمند تا قبل از تصمیم نهایی می‌تواند ویرایش (بازگشت به ابتدای گردش کار) یا لغو کند.
- مرخصی در کارتابل مدیر میانی، مدیر اصلی، منابع انسانی و مدیریت قابل مشاهده است.
- مدیر سایت فقط کارمندان مجموعه خودش را می‌بیند؛ مدیر کل همه را.

## گردش کار وام و خرید

انواعی که هنگام تعریف‌شان گزینه «نیاز به تایید مدیر اصلی» فعال شده (مثل وام و خرید):

```
کارمند ثبت می‌کند ← مدیر اصلی (سایت/کل) تایید/رد ← بخش مالی تصمیم نهایی
```

## فرم‌های داینامیک

بخش مالی و منابع انسانی هنگام تعریف نوع درخواست/قابلیت می‌توانند **فیلدهای فرم** را تعیین کنند (متن، عدد، تاریخ، انتخابی و...). کارمند هنگام ثبت، همین فرم را پر می‌کند و مقادیر سمت سرور نیز اعتبارسنجی می‌شوند.

## امنیت

- **JWT در کوکی HttpOnly** (نه localStorage) — access ۱۵ دقیقه‌ای + refresh ۷ روزه با چرخش؛ کوکی refresh فقط به مسیر `/api/auth` ارسال می‌شود؛ `SameSite=Lax` و در production `Secure`.
- **OTP**: هش‌شده با HMAC-SHA256 ذخیره می‌شود، ۵ دقیقه اعتبار، حداکثر ۵ تلاش، مقایسه timing-safe، حذف خودکار با TTL index.
- **Rate limiting**: محدودیت سختگیرانه روی درخواست/تایید OTP و محدودیت عمومی روی کل API.
- **باطل‌سازی نشست**: با غیرفعال کردن کاربر یا بازتولید کد یکتا، همه refresh token های او باطل می‌شوند (`tokenVersion`).
- **اعتبارسنجی ورودی** با zod در همه مسیرها + `sanitizeFilter` مونگوس علیه تزریق عملگرها.
- **helmet** برای هدرهای امنیتی، **CORS** فقط برای origin مجاز با credentials.
- **لاگ ممیزی (Audit Log)**: ورود موفق/ناموفق، ساخت/ویرایش کاربران، تعریف انواع، ثبت/تصمیم درخواست‌ها و مرخصی‌ها، گزارش‌کار و... همراه IP و User-Agent — قابل مشاهده در بخش مدیریت.
- فرانت از طریق **rewrite** درخواست‌ها را به API پروکسی می‌کند تا کوکی‌ها same-origin بمانند.

> ⚠️ در production حتماً مقادیر `JWT_ACCESS_SECRET`، `JWT_REFRESH_SECRET` و `OTP_SECRET` را با مقادیر قوی و یکتا جایگزین کنید و `COOKIE_SECURE=true` بگذارید (نیازمند HTTPS).

## اتصال پنل پیامک

در `apps/api/.env` مقدار `SMS_PROVIDER=kavenegar` و `KAVENEGAR_API_KEY` را تنظیم کنید. برای سرویس‌های دیگر (SMS.ir، ملی‌پیامک و...) کافیست یک `case` جدید در `apps/api/src/services/sms.service.js` اضافه کنید.

## ساختار پروژه

```
apps/
├── api/                  # سرور Express
│   ├── scripts/seed.js   # ساخت مدیر اولیه
│   └── src/
│       ├── config/       # env، اتصال دیتابیس
│       ├── models/       # User, Otp, AuditLog, RequestType, Request, LeaveRequest, WorkReport
│       ├── middleware/   # auth (JWT), rbac, validate (zod), rate limit, errors
│       ├── services/     # token, sms, audit
│       ├── validators/   # اسکیماهای zod هر ماژول
│       ├── controllers/  # منطق هندلرها (auth, users, logs, requestTypes, requests, leaves, reports, stats)
│       └── routes/       # فقط سیم‌کشی مسیرها به کنترلرها + middleware
└── web/                  # فرانت Next.js
    ├── middleware.js     # محافظت مسیرهای داشبورد
    ├── lib/              # کلاینت API (با refresh خودکار)، برچسب‌ها
    ├── components/       # UI مشترک + کارتابل‌ها و فرم‌های داینامیک
    └── app/
        ├── login/                    # ورود دومرحله‌ای
        └── dashboard/
            ├── admin/    # کاربران، لاگ رخدادها
            ├── finance/  # تعریف انواع، کارتابل مالی
            ├── hr/       # قابلیت‌ها، کارتابل خدمات، کارتابل مرخصی
            ├── manager/  # گزارش‌کار تیم، مرخصی تیم
            └── me/       # درخواست‌ها، خدمات، مرخصی، گزارش‌کار من
```

## API (خلاصه)

| متد | مسیر | دسترسی | شرح |
|---|---|---|---|
| POST | `/api/auth/login/request` | عمومی (rate-limited) | ارسال OTP با کد یکتا |
| POST | `/api/auth/login/verify` | عمومی | تایید OTP و ست کوکی JWT |
| POST | `/api/auth/refresh` | کوکی refresh | تمدید نشست |
| GET | `/api/auth/me` | ورود | اطلاعات کاربر جاری |
| GET/POST/PATCH | `/api/users...` | ADMIN | مدیریت کاربران، کد یکتا، فعال/غیرفعال |
| GET | `/api/logs` | ADMIN | لاگ همه اتفاقات |
| GET/POST/PATCH/DELETE | `/api/request-types` | FINANCE/HR/ADMIN | تعریف انواع درخواست و قابلیت‌ها |
| POST/GET/PATCH | `/api/requests...` | کارمند / بخش مربوطه | ثبت، پیگیری، لغو، تایید/رد |
| POST/GET/PATCH | `/api/leaves...` | کارمند / MANAGER / HR | گردش کار مرخصی |
| POST/GET/PATCH | `/api/reports...` | کارمند / MANAGER | گزارش‌کار |
| GET | `/api/stats` | ورود | آمار داشبورد بر اساس نقش |
